Here’s a Spring Boot rule of thumb
DTOs(A Data Transfer Object ) - Mostly public properties. Because their purpose is to transport data.
Dtos are to transfer data between application layers,
primarily between the REST Controller and the Service Layer

Rule of thumb
When you’re unsure, ask:
Is this object’s purpose to transport data, or to perform behavior?
Transport Data
data class UserResponse(...)
data class LoginRequest(...)
data class LoginResponse(...)
Usually: public val

Perform Behavior
class UserService(...)
class JwtService(...)
class PasswordValidator(...)
Usually: private val

DTO - public || Service - private

 Checked Exception (Exception)
 public void getUser() throws Exception {
     throw new Exception("User not found");
 }
 Now anybody calling it must do:
 try {
     getUser();
 } catch (Exception e) {
     ...
 }
 Java compiler forces you. These are called:Checked Exceptions

 RuntimeException
 public void getUser() {
     throw new RuntimeException("User not found");
 } Caller can simply do:
 getUser();  No try/catch required. | No throws required. | Compiler doesn’t complain.
 These are called: Unchecked Exceptions

 Why Spring prefers RuntimeException
 fun deleteUser(email: String): String {
     val user = userRepository.findByEmail(email)
         ?: throw UserNotFoundException("Email doesn't exist")

     userRepository.delete(user)

     return "Deleted"
 }
 If UserNotFoundException extended Exception, suddenly:
 fun deleteUser(...) would need: @Throws(UserNotFoundException::class) or callers would need try/catch.

 A Good Rule of Thumb Checked Exceptions Use when: Caller can reasonably recover.
 IOException
 ParseException
 FileNotFoundException

 Runtime Exceptions Use when: The request cannot continue normally.
 UserNotFoundException
 InvalidPasswordException
 UnauthorizedException
 What would the service do?  User not found. There’s no recovery inside the service. 404 Not Found
